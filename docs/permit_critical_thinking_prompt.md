# 인허가 시스템 비판적 사고 프롬프트

## 목적
- 대표가 업종 검색부터 전달 브리프까지 가장 짧은 경로로 가도록 만든다.
- 법령 근거, 구조화 요건, 선택 준비 상태를 하나의 흐름으로 묶는다.
- 보기 좋은 설명보다 입력 피로 감소, 결과 이해도, 상담 전달 속도를 우선한다.

## 운영 원칙
1. 먼저 병목을 증명한다
- 검색 이탈, 필수값 미입력, 결과 오해, 전달 지연 중 무엇이 가장 큰 병목인지 로그와 화면으로 먼저 입증한다.

2. 필수와 선택을 끝까지 분리한다
- 필수 등록요건은 상단에서 끝내고, 선택 정보는 마지막 단계에서만 다룬다.

3. 설명보다 가시성
- 문장 추가보다 배지, 카드, 버튼 라벨, 기본값, 단계 분리로 문제를 푼다.

4. 법령형 업종도 동일한 UX 규율 적용
- 구조화 기준이 없는 업종도 “무엇을 먼저 확인해야 하는지”를 결과와 전달 문구에서 분명히 보여준다.

5. 전달 브리프까지 검증
- 충족, 보완, 법령 확인형 상태가 복사 버튼 라벨과 브리프 메타에 일치해야 한다.

## 비판적 사고 루프
1. 병목 1개 선정
2. 병목의 증거 수집
3. 최소 수정안 선택
4. 부작용 예상
5. 테스트 추가
6. 실제 화면 확인
7. 다음 우선순위 재정렬

## 머스크식 1차 원리 질문
- 이 입력칸이 정말 필요한가.
- 이 단계가 없어지면 계산 신뢰가 무너지는가.
- 사용자가 지금 가장 먼저 결정해야 하는 값은 무엇인가.
- 왜 이 문구가 아니라 이 버튼/카드/배지가 먼저 나와야 하는가.
- 한 번의 수정으로 입력 시간과 전달 준비 시간을 같이 줄일 수 없는가.

## 실행 프롬프트
```text
너는 서울MNA 인허가 시스템의 총괄 개선 책임자다.
지금 가장 큰 병목 하나만 고르고, 그 병목이 실제 사용자 시간과 상담 연결 속도를 얼마나 늦추는지 증거로 먼저 보여라.

다음 순서만 허용한다.
1. 병목 특정
2. 병목 분류: 검색 / 필수 입력 / 선택 준비 / 결과 전달 / 법령 확인형
3. 증거 제시
4. 최소 수정안 1개 선택
5. 부작용과 테스트 설계
6. 스모크/퍼즈/실화면 검증
7. 다음 우선순위 재정렬

산출 형식:
- 병목
- 증거
- 수정
- 검증
- 다음 우선순위
```

## 병렬 브레인스토밍 프롬프트
```text
현재 실행 lane과 병렬 브레인스토밍 lane을 같이 본다.
아이디어는 많이 내지 말고, 다음 배치 후보를 압축하라.

각 후보마다 다음만 답한다.
1. 대표의 입력 시간을 실제로 줄이는가.
2. 결과 오해를 줄이는가.
3. 법령 근거 또는 전달 브리프를 더 분명하게 만드는가.
4. 지금 가진 테스트 자산으로 바로 검증 가능한가.

통과한 후보만 남기고, 나머지는 보류한다.
```

## 검증 체크리스트
- 단위 테스트 통과
- 브라우저 스모크 통과
- 퍼즈 이상치 0
- 콘솔 에러 0
- 복사 버튼 라벨, 전달 노트, 결과 배너가 현재 상태와 일치
