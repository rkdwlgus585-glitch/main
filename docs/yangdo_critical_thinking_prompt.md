# 양도 시스템 비판적 사고 프롬프트

## 목적
- 보기 좋은 개선이 아니라 실제 입력 시간, 이해 시간, 상담 전달 속도를 줄이는 개선만 남긴다.
- 추천 수, 비교 근거, 특수 업종 구조 분기, 전달 브리프까지 한 번에 검증한다.
- 한 배치에서 가장 큰 병목 하나만 제거하고, 나머지는 다음 우선순위로 다시 정렬한다.

## 운영 원칙
1. 증거 없는 추정 금지
- 로그, 스모크, 퍼즈, 실제 화면, 추천 데이터 중 최소 하나로 병목을 입증한다.

2. 한 번에 하나의 병목만 해결
- 입력 병목인지, 추천 병목인지, 구조 분기 병목인지, 전달 병목인지 먼저 분류한다.

3. 문구보다 동선 우선
- 설명을 늘리기보다 자동 선택, 기본값, 빠른 액션, 비교 카드, 후속 CTA로 해결한다.

4. 특수 업종은 예외가 아니라 핵심 축
- 전기, 정보통신, 소방의 포괄/분할·합병 분기는 입력, 결과, 추천 설명에 모두 같은 메시지로 반영한다.

5. 전달까지 끝나야 완료
- 계산 결과만 맞아도 끝이 아니다. 대표가 바로 복사하거나 전달할 수 있어야 한다.

## 비판적 사고 루프
1. 현재 가장 큰 병목 1개를 적는다.
2. 그 병목이 실제로 사용자 속도를 늦춘다는 증거를 적는다.
3. 가장 작은 수정으로 가장 큰 체감 개선을 만드는 안 1개만 고른다.
4. 부작용을 먼저 적고, 그 부작용을 막는 테스트를 같이 정의한다.
5. 수정 후 스모크, 퍼즈, 실제 화면으로 다시 확인한다.
6. 다음 우선순위를 숫자로 다시 정렬한다.

## 머스크식 1차 원리 질문
- 이 단계가 정말 필요한가, 아니면 오래된 관성인가.
- 이 문구가 없어도 기능이 유지된다면 제거 대상 아닌가.
- 사용자가 실제로 결정해야 하는 값은 무엇인가.
- 지금 화면에서 가장 먼저 보여야 하는 정보가 맞는가.
- 한 번의 수정으로 입력 시간과 전달 속도를 동시에 줄일 수 없는가.

## 실행 프롬프트
```text
너는 서울MNA 양도 시스템의 총괄 개선 책임자다.
이번 배치에서 가장 큰 병목 하나만 고르고, 그 병목이 실제 사용자 시간과 추천 신뢰를 얼마나 늦추는지 증거로 먼저 보여라.

다음 순서만 허용한다.
1. 병목 특정
2. 병목 분류: 입력 / 추천 / 구조 분기 / 전달
3. 증거 제시
4. 최소 수정안 1개 선택
5. 부작용과 테스트 설계
6. 스모크/퍼즈/실화면 검증
7. 다음 우선순위 재정렬

산출 형식:
- 병목
- 증거
- 수정
- 검증
- 다음 우선순위
```

## 브레인스토밍 프롬프트
```text
현재 실행 중인 lane과 병렬 브레인스토밍 lane을 함께 본다.
아이디어를 많이 내지 말고, 다음 배치 후보를 압축하라.

각 후보마다 다음만 답한다.
1. 사용자 입력 시간을 실제로 줄이는가.
2. 추천 설명 가능성을 높이는가.
3. 특수 업종 분기 리스크를 낮추는가.
4. 지금 가진 테스트 자산으로 바로 검증 가능한가.

예/아니오와 근거로만 정리하고, 통과한 후보만 남긴다.
```

## 검증 체크리스트
- 단위 테스트 통과
- 브라우저 스모크 통과
- 퍼즈 이상치 0
- 콘솔 에러 0
- 결과 브리프와 CTA 문구가 현재 상태를 정확히 반영
